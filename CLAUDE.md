# 项目开发准则与 AI 行为规范

## 1. 代码风格一致性 (Code Style Consistency)
- **首要原则**：在编写或修改任何代码前，必须先读取当前文件和相关模块的现有代码。
- **完全模仿**：严格保持与现有代码相同的缩进规则、命名规范（驼峰/蛇形等）、注释风格和架构模式。
- **局部修改**：只修改实现功能所必需的代码，除非我明确要求，否则绝对禁止为了“优化风格”而重构未涉及的代码。

## 2. 语法与质量检查 (Syntax & Error Checks)
- **主动校验**：在完成代码修改后、提交或结束任务前，你必须主动在终端执行当前项目的语法/类型检查命令（请自行读取 `package.json`、`Makefile` 或相关配置文件推断检查命令，如 linter 或 type checker）。
- **自我修复闭环**：如果检查工具返回任何错误或警告，你必须读取终端的错误输出，并自动修改代码修复这些问题。此过程需循环进行，直到没有任何语法或规范错误为止。

## 3. 依赖复用优先 (Dependency Prioritization)
- **先查后写**：在实现新功能时，必须先检查项目的依赖清单（如 `package.json`, `requirements.txt`, `build.gradle` 等）以及项目中已有的 `utils/` 或 `helpers/` 目录。
- **避免造轮子**：最大化利用已安装的第三方库和项目内已有的工具函数来解决问题，而不是自己写一套新的实现。
- **限制新增依赖**：除非找不到任何现成的替代方案，且经过我的明确同意，否则禁止私自执行命令安装新的第三方依赖。
